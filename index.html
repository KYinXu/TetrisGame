Tetris

by: Kyle Yin Xu

This game was made for my level 2 project at The League of Amazing Programmers.

1. Why I Made This Game

I decided to create this game because Tetris is one of my favorite arcade games and the one that best fits the type of game I would create. Unlike other types of games, Tetris has a basic concept that is easy to follow but difficult to master. The basic code allows it to be both enjoyable and simplistic at the same time.

2. What I Learned

By the time user testing began, I had already bug fixed for multiple weeks and found fixes to all of them beforehand. From user testing, the only feedback was to add multiple songs to prevent the game from being repetitive and boring.

3. Architecture of My Game

The game is split into three primary parts, the object, the grid, and the panel. The grid is a 2D array that keeps track of filled and unfilled blocks. Each coordinate on the array represents a square on the grid, and each square has a boolean set to either true, for filled, or false, for empty. The grid ensures that the object does not move off the screen or into filled spaces, and constantly updates to clear lines. The object is a representation of the active piece. It contains four different squares which each set their own coordinates to true on the grid. The object class controls movement and rotation of the piece. The panel is everything regarding the function of the game outside the grid. The panel keeps track of many things, such as score and level. Additionally, when a piece is held onto for later, the panel keeps track of it, as well as recording the next piece available. Additionally, the panel plays audio, sets a timer that serves as the base for the entire game, and detects commands in order to move the object itself.

4. Technical Implementation

The grid is the most important implementation in the game, and allows for everything around it to work together perfectly. It is a 2D array that keeps track of every coordinate, and sends important information to both the object and panel. The grid works by constantly updating with new information in order to keep the entire game progressing forwards. The panel has a timer that constantly sets off, providing a constant update speed for the grid, which helps it process the information to move the piece down at certain time intervals. Additionally, the grid is prompted to update whenever the panel detects any commands pressed. For example, if the player presses the left button, the panel will detect it and force the grid to update instantly, ignoring the timer. The grid is then able to update immediately and move the piece's coordinates exactly one unit to the left with little to no delay.

5. Most Challenging Part

The most challenging part of the project was managing to rotate each piece without bugs. Since the grid was on a 2D array and each piece represented four filled in squares, the only way to rotate the piece was to manually move each individual square. This meant that each block had four separate sets of commands to rotate in the four directions possible. Additionally, the rotating pieces would sometimes faze into walls and create an error. To prevent this, I added a method that would manually check whether or not the rotation would be outside the grid in order to make sure the piece would always stay inside.

6. Future Fixes and Updates

If I were to continue working on this game, I would improve the UI and layout of the game. The screen can be cluttered with too much information at the time and organizing it would help make the game easier to process. Additionally, I would create colors for individual pieces to make the pieces more easy to see.
